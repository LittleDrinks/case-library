# 产品需求文档

本文档定义 alpha 阶段的产品需求。工程架构、环境和质量门禁见
`docs/project.md`；API 现状见 `docs/api.md`；AI 约束见 `docs/ai.md`；
具体提示词、模板和分类规则见 `product_prompts/`。

## 产品定位

本产品是上海大学专版思政案例库，面向教师完成案例正文整理、AI 辅助审核、
Word 式只读批注查看、版本快照归档和一级人工审核。

它不是在线 Word，不是豆包替代品，也不是通用聊天助手。核心差异价值是：
私有化部署、案例材料归一化、审核提效、版本留痕和入库流程闭环。

## 用户角色

- **教师**：创建案例正文，粘贴来源材料，发起 AI 审核，查看和复制历史版本，
  在本地或输入页修改后重新提交。
- **审核员/管理员**：审核某一个提交版本，添加段落级批注，通过入库或退回修改。
- **公开访客**：浏览已通过案例及其来源材料。公开页面不展示 AI 批注或审核批注。

## 核心对象

系统的核心对象是**正式案例正文**。“归一化材料”不是另一个对象，而是正文的
性质：教师或 AI 将原始材料重新整理、改写、模板化，形成可提交入库的正式案例。

一个案例包含：

- 当前标题和元数据
- 正式案例正文
- 来源材料文本
- 类型和主题分类
- 教师可见的版本快照
- 绑定到版本的 AI 审核批注
- 绑定到版本的人工审核批注
- 审核和发布状态

MVP 阶段，来源材料使用纯文本框承载，概念上应填写，但不阻塞提交。教师可以粘贴
链接、公众号正文、新闻摘录、活动记录或其他纯文本。文件上传暂不支持。

## Alpha 流程

1. 教师进入案例创建输入页。
2. 教师填写或粘贴正式案例正文和来源材料。
3. 教师选择一个或多个案例类型/主题，也可以留给 AI 或审核员建议。
4. 教师点击 **AI 审核**。
5. 系统创建新的只读版本快照，如 `v1`、`v2`、`v3`。
6. AI 审核页以只读形式展示该版本正文。存在批注的段落高亮，右侧显示类似 Word
   批注的气泡。
7. 教师可以忽略 AI 批注，也可以复制任意历史版本，在本地或输入页修改后再次发起
   AI 审核，生成新版本。
8. 教师选择一个版本提交人工审核。
9. 审核员/管理员查看该版本，必要时添加段落级批注，然后通过或退回。
10. 若退回，人工批注保留在被退回版本上。教师复制正文，修改后再生成新版本。
11. 若通过，案例进入公开案例库。公开展示正式案例正文和来源材料，不展示批注。

## 版本规则

- 教师点击 AI 审核时创建新版本。
- 版本对教师可见，且只读。
- 历史版本可以打开和复制，但不能直接编辑，也不需要删除。
- 不做回滚。若 `v3` 是废稿而教师想用 `v2`，则复制 `v2`，修改后生成 `v4`。
- 版本对比暂不做，作为后续需求考虑。
- AI 批注和人工批注一旦绑定到某个版本，即保持只读和不可变。

## AI 审核范围

AI 审核仅供参考，不阻塞提交。它的作用是帮助教师自查，并提升审核员/管理员效率。

AI 应检查：

- 来源支撑缺失或较弱
- 事实、时间、人物、数字、成效表述模糊
- 上海大学元素不明确
- 所选类型/主题与正文不匹配
- 根据 `product_prompts/` 中提示词和模板判断出的结构缺项
- 课堂可用性问题，例如讨论题较弱、教学价值不清晰
- 影响清晰度、具体性或课堂可用性的语言问题

AI 不应：

- 声称自己拥有官方通过或退回权限
- 把自己包装成事实真伪裁决器
- 阻塞提交
- 默认整篇重写正文
- 在浏览器侧调用模型或暴露 API Key
- 提出没有明确理由的风格化修改要求

Alpha 阶段的事实检查应表述为“来源支撑与一致性检查”，而不是自动判断真假。
可接受的结果标签包括：“缺少来源”“来源不足”“疑似冲突”“需人工确认”。

## 批注模型

AI 批注和人工批注共用同一套产品模型。界面应接近 Word 批注：

- 只读正文中某个段落或短文本被高亮
- 右侧边栏出现批注气泡
- 批注气泡在视觉上指向对应高亮文本
- 批注绑定到具体版本

MVP 使用段落级批注。精确到字符范围的批注暂不做，因为正文一旦被修改，字符偏移
很容易失效。

后端 AI 审核应实现为内部 tool-like contract，返回结构化 JSON，而不是一整段
自由 Markdown。MVP 不要求引入 MCP；稳定的内部契约即可：

```json
{
  "comments": [
    {
      "paragraph_id": "p3",
      "quote": "可选的原文短摘录",
      "category": "source | fact | structure | classification | classroom | clarity",
      "severity": "info | suggestion | important",
      "message": "批注气泡中展示的意见",
      "suggestion": "可选的具体修改方向"
    }
  ],
  "summary": {
    "strengths": ["可选"],
    "risks": ["可选"],
    "suggested_next_steps": ["可选"]
  }
}
```

首版 UI 可以不使用严重程度筛选，但契约中保留 `severity`，方便后续扩展。

## 分类与模板

现有 `product_prompts/` 是领域提示词、模板和分类规则的来源。本文档不复制完整 prompt 逻辑。

案例类型支持多选：

- `TYPE_A`：思政课教学案例，侧重“这个材料能教什么”
- `TYPE_B`：课程思政共享资源案例，侧重专业课程与思政融合
- `TYPE_C`：实践育人案例，侧重“做了什么”及其实践育人价值

原始产品提示词资产暗示同一来源材料可能生成多个案例角度，尤其是 `TYPE_A` + `TYPE_C`。
MVP 先支持一个正式案例正文绑定多个类型标签。后续可考虑“一份来源材料生成多个
正式案例正文”。

模板符合度在 alpha 阶段是软约束：

- 用户可以提交不完美正文
- 前端不因模板缺项阻塞提交
- AI/审核员批注指出缺失或较弱的模板要素
- 最终是否通过由审核员/管理员决定

## 公开案例库

已通过案例应展示：

- 正式案例正文
- 标题和元数据
- 类型/主题标签
- 来源材料文本或来源引用

已通过案例不展示：

- AI 批注
- 人工审核批注
- 被退回版本历史
- 内部 prompt 或模型元数据

## 品牌与可替换性

Alpha 以上海大学专版优先。默认 UI 应保留当前上海大学页眉、字标、导航和红色
学术视觉语言。设计参考包括 `docs/design/create/*.png` 和本地
`docs/case-library-design.zip`。

开源可替换性的含义是：其他学校应能替换品牌、标题、API 设置、提示词和模板，而
不需要重写产品代码。学校专属提示词资产可以继续放在 `product_prompts/`。

## Alpha 非目标

- 流式聊天
- 会话切换
- 富文本在线 Word 编辑
- 文件上传
- Word/PDF 导出
- 版本 diff
- 回滚
- 多级审核
- AI 自动通过/退回 gating
- AI 率检测
- 自动事实真伪裁决
- 公开展示审核批注

## 技术风险

- **批注锚定**：句子级或字符级偏移会在正文修改后漂移。Alpha 使用段落级批注。
- **AI 响应稳定性**：必须要求结构化 JSON，并在服务端校验。不要把任意 Markdown
  当作产品状态解析。
- **版本数据模型**：正文、来源材料、AI 批注、人工批注和状态都必须感知版本。
  单一可变的 `case.content` 字段不够。
- **Prompt 注入**：用户粘贴的来源材料不可信。提示词必须把用户文本视为待检查材料，
  而不是系统指令。
- **事实检查过度承诺**：产品文案不能承诺自动判断真假。
- **范围膨胀**：Word 式编辑、导出、diff、文件解析、多正文生成都是后续能力，不是
  alpha 必需项。
- **开源泄密**：不得提交密钥、私有服务地址或硬编码 API Key。品牌和 prompt 应可替换。

## 初始验收标准

- 教师可以粘贴案例正文和来源材料。
- 教师可以发起 AI 审核，并创建只读版本。
- 审核页可以展示段落高亮和右侧批注气泡。
- 教师可以查看并复制历史版本。
- 审核员/管理员可以审核一个版本，并添加段落级批注。
- 审核员/管理员可以通过或退回该版本。
- 已通过公开案例展示正文、元数据、标签和来源材料，但不展示批注。
- AI 审核在服务端执行，受功能开关控制，并返回结构化 JSON 批注契约。
